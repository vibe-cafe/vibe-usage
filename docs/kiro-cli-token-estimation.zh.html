01

提交 858336b 读 data.sqlite3 里的 conversations_v2 / conversations,外加 ~/.kiro_sessions/*.json 归档。在一台真实、重度使用的机器上:

现有路径读取的位置测试机行数
data.sqlite3 → conversations_v20
data.sqlite3 → conversations0
~/.kiro_sessions/*.json目录不存在

真实数据在 ~/.kiro/sessions/cli/*.jsonl(测试机两天内 116 个文件被改),是事件流格式,而非 DB 路径期待的 history[] 结构。

02
原生格式

<uuid>.jsonl — 每行一个事件 {version, kind, data}:

kind载荷对应
Promptcontent[](text/image);meta.timestamp = epoch 秒新增输入 · 唯一时间戳
AssistantMessagecontent[]:text · thinking(text + modelId + 加密 signature)· toolUse(name + input)输出 / 推理 · 模型 id
ToolResultscontent[] 工具输出下一轮的新增输入
Compactionsummary重置累计上下文

同名 <uuid>.json:cwd → 项目;session_state.rts_model_state.model_info.model_id → 模型回退。

03
估算

token = 字符串叶子字符数 / 4,只遍历字符串叶子,并跳过非语言键(signature, redactedContent, toolUseId, modelId, message_id, format, id)。每个 assistant 轮次:

outputTokens          = assistant text + toolUse(name/input)
reasoningOutputTokens = thinking.text            (排除签名)
inputTokens           = prompt + 之前的 ToolResults (每图 +~1600)
cachedInputTokens     = 累计上下文(每轮重发) + 系统提示开销
model    = thinking.modelId -> 会话 model_id -> 'kiro-token-estimate'
timestamp = 所在 Prompt 的时间戳(同轮继承)
正确性陷阱 · 排除签名

扩展思考块带一段 ~500 字符的加密 signature。在测试机上约为 148 万 token 的非语言字节。数整段 JSON(或"所有字符串值")会让 output 虚高 >100%。输出用 assistant 文本 估算,也比数 chunk 更贴近真实(此原生格式里没有 time_between_chunks)。

04
数字是怎么验证的

两份独立编写的计数器跑同一批日志:估算器(有状态轮次遍历 + 选择性提取) vs. 单遍暴力累加所有字符串叶子(无轮次、无 cache、不跳字段)。

  • 抓到真 bug:早期版本用 JSON.stringify 提取工具结果 → 结构开销 → 6 倍虚高(输入 1480万 vs 250万)。只有独立实现才暴露得出。
  • 收敛:修复后输入对账到 99.5%(2,532,691 vs 2,544,964)。
  • 差异可解释:输出侧差值恰等于第三次独立测得的签名量(~148万)—— 刻意排除,非 bug。

范围:验证解析正确性与一致性,不验证 chars/4 比例、也不验证 cache 模型是否等于厂商计费(Kiro 不暴露 ground truth,按 Credits / INVOCATIONS 计量,与 token 非线性)。

参考:该解析器在贡献者机器上跑出——新增(in+out+think)≈ 3.67M,cacheRead ≈ 662M,主力模型 claude-opus-4.8。

05
两个"每轮重发"项 — 你定夺

(A) thinking 签名进 cache

thinking 块(含签名)后续每轮都会被重发,所以签名加进累计上下文(不计入 output)。按 chars/4 计,是保守下限(base64 分词更差)。

(B) 系统提示 + 工具 schema

每次请求注入但从不写日志,所以事件流一个都数不到。建模为每轮常量 KIRO_CLI_SYSTEM_OVERHEAD_TOKENS(默认 20000)。设为 0 即只数日志内文本。随你改 / 删。
