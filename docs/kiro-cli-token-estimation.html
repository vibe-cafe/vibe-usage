Kiro CLI token estimation
read the native event stream

The shipping Kiro CLI records conversations as a {version, kind, data} event stream — not in conversations_v2. This note documents the new parser, the estimation model, two reviewer decision points, and how the numbers were cross-validated.

① Why the DB path is empty ② Native format ③ Estimation ④ Validation ⑤ Decision points
TL;DR
Adds ~/.kiro/sessions/cli/*.jsonl as the first Kiro source; estimates in/out/thinking/cache from text (chars/4); falls back to the existing credit / legacy paths. Cross-validated — input reconciles to 99.5%. Key trap handled: thinking-block signatures must be excluded from token counts (~1.5M tokens of noise, would inflate output >100%).
01
Why the current CLI path finds nothing

Commit 858336b reads conversations_v2 / conversations from data.sqlite3, plus ~/.kiro_sessions/*.json archives. On a real, heavily-used install:

Source the current path readsRows on the test machine
data.sqlite3 → conversations_v20
data.sqlite3 → conversations0

The real data lives in ~/.kiro/sessions/cli/*.jsonl (116 files touched in 2 days on the test machine), in an event-stream format — not the history[] shape the DB path expects.

02
The native format

<uuid>.jsonl — one event per line, {version, kind, data}:

kindpayloadmaps to
Promptcontent[] (text/image); meta.timestamp = epoch secondsfresh input · only timestamp
AssistantMessagecontent[]: text · thinking (text + modelId + crypto signature) · toolUse (name + input)output / reasoning · model id
ToolResultscontent[] tool outputfresh input to next turn
Compactionsummaryresets running context

Companion <uuid>.json: cwd → project; session_state.rts_model_state.model_info.model_id → model fallback.

03
Estimation

token = string-leaf-chars / 4, walking only string leaves and skipping non-linguistic keys (signature, redactedContent, toolUseId, modelId, message_id, format, id). Per assistant turn:

outputTokens          = assistant text + toolUse(name/input)
reasoningOutputTokens = thinking.text            (signature excluded)
inputTokens           = prompt + preceding ToolResults (+~1600/image)
cachedInputTokens     = running context (re-sent each turn) + system overhead
model    = thinking.modelId -> session model_id -> 'kiro-token-estimate'
timestamp = enclosing Prompt timestamp (turns inherit it)
Correctness trap · exclude signatures

Extended-thinking blocks carry a ~500-char crypto signature. On the test machine that is ~1.48M tokens of non-linguistic bytes. Counting the raw JSON (or "all string values") inflates output by >100%. Output is estimated from assistant text, which is also more faithful than a chunk count (time_between_chunks is not present in this native format).

04
How the numbers were validated

Two independently-written counters over the same logs: the estimator (stateful turn walker, selective extraction) vs. a dead-simple single pass summing every string leaf (no turn logic, no cache, no key-skipping).

  • Caught a real bug: an early version extracted tool results via JSON.stringify → structural overhead → 6× inflation (14.8M vs 2.5M input). Only independent implementations surface that.
  • Converged: after the fix, input reconciles to 99.5% (2,532,691 vs 2,544,964).
  • Gap fully explained: the output-side difference equalled a third, separate measurement of signature volume (~1.48M) — a deliberate exclusion, not a bug.

Scope: validates parsing correctness & consistency — not the chars/4 ratio or that the cache model matches provider billing (Kiro exposes no ground truth; it meters in Credits / INVOCATIONS, non-linear in tokens).

For reference, the parser on the contributor's machine: fresh (in+out+think) ≈ 3.67M, cacheRead ≈ 662M, dominant model claude-opus-4.8.

05
Two "re-sent every turn" terms — your call

(A) Thinking signatures in cache

Thinking blocks (incl. signature) are re-sent on every later request, so the signature is added to the running context (never to output). Counted at chars/4 — a conservative floor (base64 tokenizes worse).

(B) System prompt + tool schemas

Injected on every request but never logged, so the stream counts none of them. Modeled as a per-turn constant KIRO_CLI_SYSTEM_OVERHEAD_TOKENS (default 20000). Set to 0 to count only logged text. Happy to tune/remove.

Tests

  • cliEventsToEntries — estimation, signature excluded from output but kept in cache, structure-not-inflated counting, prompt-timestamp inheritance.
  • parse — reads a native *.jsonl fixture end-to-end. All existing Kiro tests still pass.
